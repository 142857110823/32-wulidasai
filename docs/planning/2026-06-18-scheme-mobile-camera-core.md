# 手机拍照核心问题方案

## 当前进度

- 已确认项目当前仍以模拟图像与模拟流程为主。
- `pubspec.yaml` 目前未体现真实 camera / permission 相关依赖已正式接入。
- Android 侧未确认存在完整相机权限声明与授权链路。
- 采集页面当前仍以 demo / mock 入口承担主流程演示。

## 待完成工作

- 建立真实手机拍照能力的最小闭环。
- 让采集主链从“模拟图片入口”升级为“真实预览 + 真实拍摄 + 可回退模拟模式”。
- 保持 UI 主链不被真实接入打散。

## 执行步骤

1. 建立相机能力边界。
   只做 Android 优先最小闭环，不同时扩 iOS / 桌面 / Web 的真实拍照能力。

2. 接入权限链。
   明确相机权限声明、授权请求、拒绝后的阻断提示与回退路径。

3. 接入真实预览与拍摄。
   在采集主链里形成真实预览、拍摄 I0、拍摄 I1 的最小可用链路。

4. 保留模拟模式并行存在。
   真实拍照接入后，不删除模拟模式，因为模拟模式仍承担答辩演示和平台点击验证职责。

5. 与主链步骤对齐。
   真实拍照能力必须严格嵌入 `质控 -> I0 -> I1 -> ROI -> 特征 -> 预测 -> 保存`，不能另起一套分叉流程。

## 验证标准

- Android 设备上可以拉起真实相机或真实相机预览。
- 用户可完成 I0 与 I1 的真实拍摄或确认。
- 权限拒绝时有明确阻断反馈，不假装成功。
- 模拟模式仍可独立运行，且有清晰标记。
- UI 主链没有因真实拍照接入而退化成杂乱多入口。

## 验收条件

- 存在真实拍照入口、真实授权链、真实预览/拍摄闭环。
- 真实拍照已进入主链，而不是只停留在底层适配代码。
- 浏览器或设备可见的采集页语义从“模拟演示”提升到“可真实采集”。

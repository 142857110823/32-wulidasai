# FreshSalt Surface 项目熟悉记录

## 1. 项目定位判断

我对这个 APP 的判断很明确：它不是“测盐神器”，不是食品安全判断器，也不是执法检测工具。

它的真实定位是：

- 一个面向大学物理竞赛的 Android 优先 Flutter 原型平台。
- 一个把“受控 RGB 成像 + ROI / 灰卡质控 + 灰箱模型 + 模拟验证”组织成完整产品主链的实验型 APP。
- 一个先完成平台与交互闭环、再逐步替换真实相机与真实图像处理的工程容器。

这意味着后续所有工作都不能偏到“功能越多越好”，而要始终围绕三件事：

- 平台是否像一个完整平台。
- 主链是否清晰可点通。
- UI 是否持续表达“模拟验证平台”而不是“最终检测结论”。

## 2. 当前项目状态判断

基于已读规则、基准文档、代码结构和浏览器截图，我认为当前项目已经完成了“从零骨架到可见平台雏形”的关键跨越，但仍然没有进入“核心能力完成”阶段。

当前可以成立的判断：

- 首页平台骨架已经基本形成。
- 结果详情页已经从空壳/空白推进到可见结果页。
- 模拟模式、模型包、历史、报告、分析、点击验证等外围模块都已有雏形。
- 项目在“演示平台”层面已经比纯路由骨架前进很多。

当前不能成立的判断：

- 不能说已经具备真实手机拍照能力。
- 不能说已经具备真实图片处理能力。
- 不能说 UI 已完全对齐设计图。
- 不能把“页面存在”“结果能显示”误判为“平台已完成”。

## 3. 我对产品主链的理解

这个项目真正应该被用户感知到的主链不是“进入很多页面”，而是：

`首页 -> 质控 -> I0 -> I1 -> ROI -> 特征 -> 预测 -> 保存 -> 结果复核`

我认为这条主链有三个要求：

- 要像实验测量流程，而不是像文档导航。
- 要让用户始终知道自己当前在链路的哪个阶段。
- 要让每一步都服务于下一步，而不是成为独立说明卡片。

如果这条链不清晰，那么即使模块很多，也只是“功能堆叠”，不是平台。

## 4. 我对 UI 的核心看法

当前 UI 最大的积极点：

- 已经开始从“松散功能页”收敛成“平台工作台”。
- 首页第一屏已接近完整平台观感。
- 结果页已经具备主结果、范围、趋势、解释的基本叙事框架。

当前 UI 最大的问题不是“颜色不够好看”，而是三个更本质的问题：

- 采集主链的测量语义仍然不够强。
- 首页与结果页之间的平台语言还不够完全统一。
- 浏览器可见结果虽然稳定很多，但仍有细节级偏差，没有达到彻底放心验收的状态。

我认为这个项目后续 UI 工作不能按“补页面”推进，而要按“补平台观感”推进。

## 5. 我对技术结构的看法

这个项目当前技术结构方向是对的，尤其是分层意图基本清楚：

- 页面层负责展示、导航、状态表达。
- 服务层负责业务流程。
- orchestrator 负责串链。
- repository 负责结果与记录。
- demo / mock 资源负责首阶段演示验证。

但我也认为，当前技术结构存在一个重要现实：

- 平台 UI 可见度已经跑在真实能力前面。

这不是坏事，反而符合基准文档“先平台、后实验、再模型”的路线。
真正的问题不在于“为什么现在还是假的”，而在于：

- 模拟边界有没有持续说清。
- 真能力接入点有没有预留清楚。
- 后续从 mock 到 real 的替换是否会冲掉现有主链。

## 6. 我对当前三大核心问题的判断

### 6.1 手机拍照能力

当前没有真实 camera 依赖、没有 Android 相机权限、没有真实预览/拍摄链路。

所以这项不是“待优化”，而是“尚未开始实装完成”。

### 6.2 真实图片处理能力

当前特征、质控、ROI、差分结果仍然主要依赖 metadata / demo 注入链路。

所以这项也不是“精度不够”，而是“真实像素处理链尚未接入”。

### 6.3 UI 是否符合设计图

这是三项里最接近完成的一项，但仍不能判定为完成。

原因不是因为页面不可见，而是因为：

- 采集主链仍不够像测量平台。
- 局部页面仍有解释性偏重的问题。
- 与基准图相比，仍有平台叙事密度和实验语义的差距。

## 7. 我认为后续工作的正确顺序

我不建议把三大核心混成一个大修复回合。
正确顺序应当是：

1. 继续用浏览器可见结果修 UI，且一次只打一个最高优先级可见偏差。
2. 在 UI 主链基本稳住后，再进入真实手机拍照接入。
3. 手机拍照入口成立后，再接真实图像处理。
4. 最后再用最小必要验证补强，而不是提前跑很多无关测试。

这个顺序符合：

- 用户当前把 UI 放在最高优先级。
- 基准文档强调先平台再实验。
- 当前代码现实是 mock 平台壳已先行，真实能力尚未接入。

## 8. 我认为最重要的工作纪律

后续如果继续推进，我认为必须守住以下纪律：

- 永远先看浏览器里真实可见的页面，再看代码。
- 没有首页/主链/结果页的明显可见进步，就不要转去补次要模块。
- 每次只修一个最高优先级偏差。
- 任何偏差先写入 `outputs/workflow-error-2026-06-15.md`，再继续。
- 不用 `flutter analyze` 或 `flutter test` 替代前端验收。

## 9. 我的总体结论

这个项目现在最有价值的地方，不是“功能多”，而是已经具备了继续收敛成完整竞赛平台的基础。

我的结论是：

- 项目方向是对的。
- 平台路线是对的。
- UI 已经进入可持续打磨阶段。
- 三大核心里，UI 是当前最该继续打的主战场。
- 手机拍照与真实图像处理必须后续分阶段单独推进，不能假装它们已经具备。

如果只用一句话概括：

`FreshSalt Surface` 现在已经像“一个正在成型的平台”，但还不是“一个三大核心都完成的平台”。

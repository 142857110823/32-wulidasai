# UI 核心问题方案

## 当前进度

- 已完成工作区规则、基准文档、浏览器截图和当前 Flutter 页面结构的核对。
- 首页第一屏已经形成“今日任务 + 核心入口 + 平台状态 + 最近模拟趋势”的平台骨架。
- 结果详情页已经能稳定展示主结果、范围标尺、趋势块、模型输入摘要与模型解释。
- 当前 UI 已明显优于纯骨架阶段，但仍未达到“完全对齐基准图”的验收结论。

## 待完成工作

- 继续缩小浏览器可见 UI 与基准图之间的差异。
- 重点补强采集主链的实验测量语义，而不是继续扩页面数量。
- 统一首页、采集链、结果页之间的视觉语言与信息密度。

## 执行步骤

1. 固定验收面。
   使用稳定可见的浏览器验收入口，只以真实页面截图作为 UI 判断依据。

2. 锁定最高优先级页面。
   在首页、采集主链、结果页三者中，只选当前最影响平台观感的一个可见偏差处理。

3. 优先补主链语义，不补外围模块。
   若首页已基本成型，则下一个重点应转向采集主链页面是否体现相机预览、ROI、灰卡、质控的实验语义。

4. 保持结果页为第二优先级复核面。
   每完成一次首页或主链调整，都要回看结果页是否仍然保持统一的视觉节奏和平台感。

5. 严格一轮一偏差。
   每次只修一个最高优先级可见问题，修完就回浏览器复核，不并行发散。

## 验证标准

- 首页第一屏必须一眼读出“平台状态 + 当前任务 + 主入口 + 趋势”。
- 采集主链必须一眼读出“测量流程”，而不是“说明文案集合”。
- 结果页必须稳定呈现主结果、范围、趋势、解释、边界提示五类关键信息。
- 所有模拟数据必须持续保留“模拟数据”标记。
- 浏览器里的真实可见结果必须优先于代码结构说明。

## 验收条件

- 首页、采集主链、结果页三者在浏览器中形成统一平台观感。
- 主链页面具备清晰实验语义，不再明显偏说明式布局。
- 结果页不空白、不弱化、不退回占位态。
- 当前轮次的最高优先级 UI 偏差被真实截图证明已缩小或消除。

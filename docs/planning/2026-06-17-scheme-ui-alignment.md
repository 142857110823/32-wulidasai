# UI 设计符合度方案

## 当前进度

- 已重新核对 `AGENTS.md`、`freshsalt.rules.md`、基准 markdown、基准 docx。
- 已提取 docx 的高优先级设计点：
  首页：任务入口、演示模式、最近模拟趋势。
  采集预测：相机预览、ROI、灰卡、盐晶体纹理、成像质控。
  结果详情：主结果、范围标尺、历史趋势、模型解释、边界提示。
- 已取得首页、质控页和结果页的浏览器级证据，其中结果页当前为空白。

## 待完成工作

- 用稳定浏览器证据覆盖首页与主链。
- 逐页对照基准而不是对照现有路由。
- 只围绕第一屏、主链叙事、结果页观感判断是否对齐。

## 执行步骤

1. 先看基准，再看浏览器。
   以基准 docx 和基准 markdown 为最高视觉依据，不接受“代码里有页面”作为完成证明。
2. 审查首页第一屏。
   判断是否先让用户进入任务，而不是先读大量说明文案。
3. 审查采集主链。
   判断质控、I0、I1、ROI、特征、预测是否像一个实验测量流程，而不是一组稀疏说明卡片。
4. 审查结果页。
   判断是否稳定可见，并包含主结果、范围标尺、趋势、解释、边界提示。
5. 审查统一性。
   判断首页、主链、结果页在层级、间距、文案密度、模块节奏上是否属于同一平台语言。
6. 任何可见偏差先记日志，再决定下一项。

## 验证标准

- 首页第一屏具有任务导向，不以大段解释文案占主导。
- 采集链可见地体现相机预览、ROI、灰卡和质控语义。
- 结果页稳定可见，并具备基准要求的五类信息。
- 模拟数据标记持续存在。
- 浏览器可见结果优先于代码结构证明。

## 验收条件

- 只有当“首页可见对齐 + 主链可见对齐 + 结果页可见对齐 + 整体统一性成立”同时满足，才可判定 UI 符合设计图。
- 只要结果页空白、首页过度解释、主链过于抽象或浏览器证据不稳定，就必须判定未满足。
